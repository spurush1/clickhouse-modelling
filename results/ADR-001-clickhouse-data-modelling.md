**Document type**: Architecture Decision Record  
**Status**: PROPOSED — pending board approval  
**Authors**: Purushothaman Srinivasanarasimhan

## 1. Executive Summary

This document presents the results of a controlled empirical benchmark comparing three
ClickHouse data modelling patterns for procurement data with a multi-level one-to-many
hierarchy (`order → purchase_order → purchase_order_line`). The study was motivated by a
specific constraint arising from the Scorpio NLP2SQL agent: an LLM writing SQL on the fly
cannot be expected to know which aggregation patterns avoid double-counting, to choose the
right join algorithm per query shape, or to reason about deduplication semantics on
`ReplacingMergeTree` tables. The data model must make correct queries *easy* and incorrect
queries *hard to write accidentally*.

**Bottom line for the board:**

| Pattern | Header queries | Line queries | Cross-grain | Memory at 100M | NLP2SQL safety | Verdict |
|---|---|---|---|---|---|---|
| A — Co-sorted tables + dict | Fast | **7.4s** ❌ | **6.8s** ❌ | 730 MB – 1.1 GB | Medium | Not recommended |
| B — Nested arrays | Fast | 524ms | 76ms | 61–276 MB | Low ❌ | Niche only |
| C — Flat denormalized | Fast | **268ms** ✅ | **87ms** ✅ | **9–14 MB** ✅ | **High** ✅ | **Recommended** |

## 2. Problem Statement

### 2.1 The double-counting problem

A Purchase Order header record relates to multiple line items in a 1:N relationship.
If the two entities are merged into a single flat row (one row per line), every header-level
field (`po_total`, `vendor_id`, `order_date`, etc.) is physically repeated on every line row.

Naively aggregating this table produces wrong results:

```sql
-- WRONG: sums po_total once per line, not once per PO
SELECT vendor_id, SUM(po_total) FROM po_flat GROUP BY vendor_id;

-- A PO with 10 lines will contribute 10× its total — classic double-count
```

This is not a ClickHouse-specific problem; it exists in any columnar store where
parent-child data is denormalized at the child grain. The question is: which modelling
choice makes it easiest to avoid, especially when the SQL is generated by an LLM?

### 2.2 The NLP2SQL constraint

Scorpio's NLP2SQL agent (nlp2sql_agent) translates natural language questions into SQL
and executes them against ClickHouse gold marts. The agent:

- Receives a user question in plain English
- Uses an LLM (Claude Sonnet/Opus) to synthesize a ClickHouse SQL query
- Executes the query and streams results back to the UI

This creates hard constraints on the data model:

| Constraint | Implication for data model |
|---|---|
| LLM writes SQL without runtime context | The schema itself must encode which columns are safe to SUM |
| LLM cannot call `dictGet()` reliably | Dictionary lookups require schema knowledge not in the prompt |
| LLM may not know when to use `ARRAY JOIN` | Nested arrays require explicit syntax the LLM must learn |
| LLM may not know `FINAL` is needed | `ReplacingMergeTree` returns duplicates without FINAL |
| Prompt token budget is finite | More tables = more schema tokens = higher cost per query |
| Wrong SQL must be obviously wrong | A 10× overstated revenue number must be detectable |

The ideal model for NLP2SQL is one where:
1. The simplest possible SQL (direct `SUM`, `COUNT`, `GROUP BY`) produces correct results
2. Cross-entity questions don't require join knowledge
3. A single table covers the most common analytical questions

## 4. Methodology

### 4.1 Test environment

| Resource | Specification |
|---|---|
| Machine | Intel i7-12700H (14 cores / 20 threads), 32 GB RAM, NVMe SSD |
| Docker limits | 4 CPUs / 16 GB RAM (Docker Desktop default on Windows) |
| Container | `clickhouse/clickhouse-server:24.3-alpine` (isolated, no other services) |
| ClickHouse version | 24.3.18 (official build) |
| Other services | All stopped before benchmark (Postgres, Redis, RabbitMQ, Keycloak, MinIO) |

### 4.2 Isolation protocol

A throwaway Docker container (`ch-bench`) was used exclusively for the benchmark,
separate from the production `scorpio-clickhouse` container. This ensures:
- No shared page cache with other workloads
- No background compaction or merge pressure from production data
- Reproducible starting state via `DROP DATABASE IF EXISTS bench`

### 4.3 Measurement method

- Each query run **5 times** per (pattern × tier) combination
- All 5 runs collected; **median reported** (this naturally suppresses the first cached run)
- Timings sourced from `system.query_log` (`query_duration_ms`, `memory_usage`, `read_rows`)
- Query identification via embedded comment tags: `/* bench:QN:PATTERN:TIER */`

```sql
-- Example collection query
SELECT
    extractAllGroups(query, 'bench:(Q\\d+):(\\w+):(\\w+)')[1][1] AS query_id,
    extractAllGroups(query, 'bench:(Q\\d+):(\\w+):(\\w+)')[1][2] AS pattern,
    extractAllGroups(query, 'bench:(Q\\d+):(\\w+):(\\w+)')[1][3] AS tier,
    median(query_duration_ms)              AS median_ms,
    round(max(memory_usage) / 1e6, 1)     AS peak_mem_mb,
    any(read_rows)                         AS rows_read
FROM system.query_log
WHERE query LIKE '%bench:Q%' AND type = 'QueryFinish'
GROUP BY query_id, pattern, tier
ORDER BY query_id, tier, pattern;
```

### 4.4 Correctness gate

Before benchmark timing was recorded, all five queries were validated to return **identical
results** across all three patterns at every tier. A mismatch would indicate a double-counting
bug or schema error. All 20 correctness assertions (5 queries × 4 tiers) passed.

## 5. Three Modelling Patterns

### 5.1 Pattern A — Co-sorted ReplacingMergeTree + HASHED Dictionary

**Design philosophy**: Keep normalized tables; use ClickHouse's `full_sorting_merge` join
algorithm to make joins cheap when both tables are physically sorted on the join key.
Use an in-memory dictionary for the smallest dimension (order table).

**DDL:**

```sql
-- Top-level order: sorted by order_id
CREATE TABLE order_a (
    order_id     UInt64,
    customer_id  UInt32,
    order_date   Date,
    region       LowCardinality(String),
    order_status LowCardinality(String),
    version      UInt64
) ENGINE = ReplacingMergeTree(version)
  PARTITION BY toYYYYMM(order_date)
  ORDER BY order_id;

-- Purchase order: leading ORDER BY key = order_id (same as order_a → co-sort)
CREATE TABLE purchase_order_a (
    po_id     UInt64,
    order_id  UInt64,
    vendor_id UInt32,
    po_total  Decimal(18,2),
    po_date   Date,
    po_status LowCardinality(String),
    version   UInt64
) ENGINE = ReplacingMergeTree(version)
  PARTITION BY toYYYYMM(po_date)
  ORDER BY (order_id, po_id);    -- ← co-sorted with order_a on order_id

-- PO line: leading ORDER BY key = po_id (same as purchase_order_a → co-sort)
-- order_id denormalized to skip middle join for order-grain questions
CREATE TABLE po_line_a (
    po_id       UInt64,
    line_id     UInt32,
    order_id    UInt64,           -- denormalized FK (avoids middle-table join)
    product_id  UInt32,
    quantity    UInt32,
    unit_price  Decimal(18,2),
    line_amount Decimal(18,2),
    line_date   Date,
    version     UInt64
) ENGINE = ReplacingMergeTree(version)
  PARTITION BY toYYYYMM(line_date)
  ORDER BY (po_id, line_id);     -- ← co-sorted with purchase_order_a on po_id

-- In-memory dictionary for order-level attributes (O(1) lookup per row)
CREATE DICTIONARY order_dict (
    order_id UInt64, customer_id UInt32,
    order_date Date, region String, order_status String
) PRIMARY KEY order_id
  SOURCE(CLICKHOUSE(TABLE 'order_a' DB 'bench'))
  LAYOUT(HASHED())
  LIFETIME(MIN 300 MAX 600);
```

**Query patterns:**

```sql
-- Header query — direct, no join
SELECT vendor_id, sum(po_total)
FROM purchase_order_a FINAL         -- FINAL required for correct dedup
WHERE po_date >= today() - 30
GROUP BY vendor_id;

-- Cross-grain join — sort phase skipped because both tables co-sorted on po_id
SELECT p.vendor_id, sum(l.line_amount)
FROM po_line_a AS l FINAL
JOIN purchase_order_a AS p ON l.po_id = p.po_id
GROUP BY p.vendor_id
SETTINGS join_algorithm = 'full_sorting_merge';

-- Region lookup via dictionary (avoids joining order_a entirely)
SELECT dictGet('bench.order_dict', 'region', toUInt64(order_id)) AS region,
       sum(line_amount)
FROM po_line_a FINAL
GROUP BY region;
```

**How the join algorithm works:**  
When `full_sorting_merge` is selected and both tables' physical row order matches the join
key, ClickHouse skips the sort phase entirely. Instead of building an in-memory hash table
(O(N) memory), it streams both sorted inputs block by block and merges them in-order —
analogous to a merge-sort merge step. Memory usage is O(block_size) rather than O(N).

**Critical flaw discovered in benchmark:**  
The `FINAL` keyword forces ClickHouse to perform a deduplication merge at query time even
when there are no duplicates (e.g., fresh data). This is a full table scan with an extra
merge step. At 100M lines, this adds ~7 seconds to Q2 and Q3. There is no way to avoid
`FINAL` on `ReplacingMergeTree` without accepting stale duplicate rows in results.

**NLP2SQL compatibility: Medium**
- LLM must know to append `FINAL` to every `ReplacingMergeTree` query
- LLM must specify `SETTINGS join_algorithm = 'full_sorting_merge'` for cross-grain queries
- LLM must know `dictGet()` syntax and the dictionary name
- Three separate tables increase schema token budget

### 5.2 Pattern B — Nested Arrays (one row per PO)

**Design philosophy**: Store the parent entity (PO) as a single row with child line items
packed into parallel `Array()` columns. This matches ClickHouse's native columnar storage
model for sub-entities — all line data for a PO is physically co-located on disk.

**DDL:**

```sql
CREATE TABLE purchase_order_b (
    po_id    UInt64,
    order_id UInt64,
    vendor_id UInt32,
    po_total  Decimal(18,2),      -- scalar: safe to SUM directly
    po_date   Date,
    po_status LowCardinality(String),
    -- Parallel arrays — ALL must have identical length per row
    `lines.line_id`     Array(UInt32),
    `lines.product_id`  Array(UInt32),
    `lines.quantity`    Array(UInt32),
    `lines.unit_price`  Array(Decimal(18,2)),
    `lines.line_amount` Array(Decimal(18,2))
) ENGINE = MergeTree()
  PARTITION BY toYYYYMM(po_date)
  ORDER BY (order_id, po_id);
```

**Query patterns:**

```sql
-- Header query — naturally correct, no special handling needed
SELECT vendor_id, sum(po_total) FROM purchase_order_b
WHERE po_date >= today() - 30
GROUP BY vendor_id;

-- Line query — ARRAY JOIN explodes arrays back into rows
SELECT lines.product_id, sum(lines.line_amount)
FROM purchase_order_b
ARRAY JOIN lines
GROUP BY lines.product_id;

-- Cross-grain membership — has() checks array without full explode
SELECT count(DISTINCT order_id)
FROM purchase_order_b
WHERE has(`lines.product_id`, 42);
```

**How ARRAY JOIN works:**  
`ARRAY JOIN` is a ClickHouse-specific clause that explodes one or more Array columns into
individual rows. For each row in `purchase_order_b` with N lines, `ARRAY JOIN lines`
produces N output rows with the scalar PO columns repeated and the array elements unpacked.
This is equivalent to a `LATERAL JOIN UNNEST` in Postgres, but operates natively on
ClickHouse's columnar blocks without materializing intermediate rows.

The `has(array, value)` function scans the array of each row and returns true if the value
is present. Because arrays are stored as compressed columnar blocks, this is extremely
cache-efficient — the benchmark measured `has()` at 76ms for 10M POs (Q3:B:XL),
beating Pattern C's `DISTINCT` scan at 87ms.

**Memory wall discovered in benchmark:**  
At XL tier (100M lines / 10M POs), building `purchase_order_b` via a single `groupArray`
aggregation exceeded the 14 GB container limit and OOM-crashed. This is because `groupArray`
materialises all line arrays in memory simultaneously during the GROUP BY phase.
The workaround — batching in 10 × 1M PO ranges — adds significant ETL complexity and
is not suitable for streaming CDC pipelines.

**NLP2SQL compatibility: Low**
- LLM must know `ARRAY JOIN` syntax (not in standard SQL)
- LLM must know when to use `has()` vs `ARRAY JOIN` for different query shapes
- LLM cannot use standard aggregation patterns on nested columns without ARRAY JOIN
- BI tools (Tableau, Metabase, Looker) do not understand Array columns natively

### 5.3 Pattern C — Flat Denormalized Line-Grain Table

**Design philosophy**: Pre-join all three entities at ingest time. Every row in the table
contains all fields from `order`, `purchase_order`, and `po_line`. Add a single boolean
marker (`is_first_line`) to disambiguate header-grain from line-grain aggregations.

**DDL:**

```sql
CREATE TABLE po_flat_c (
    -- Line-grain primary key
    po_id       UInt64,
    line_id     UInt32,
    -- Denormalized PO header (repeated on every line row of this PO)
    order_id    UInt64,
    vendor_id   UInt32,
    po_total    Decimal(18,2),    -- ⚠ DO NOT SUM — use sumIf(po_total, is_first_line=1)
    po_date     Date,
    po_status   LowCardinality(String),
    -- Denormalized order header (repeated on every line row of this order's POs)
    customer_id UInt32,
    order_date  Date,
    region      LowCardinality(String),
    -- Line-level fields — always safe to SUM
    product_id  UInt32,
    quantity    UInt32,
    unit_price  Decimal(18,2),
    line_amount Decimal(18,2),    -- ✅ SAFE to SUM directly
    -- Double-count guard: 1 on first line per PO, 0 on all subsequent lines
    is_first_line UInt8
) ENGINE = MergeTree()
  PARTITION BY toYYYYMM(po_date)
  ORDER BY (po_date, vendor_id, po_id, line_id);
```

**Ingest pipeline** (how `is_first_line` is computed):

```sql
INSERT INTO po_flat_c
SELECT
    l.po_id, l.line_id, l.order_id,
    p.vendor_id, p.po_total, p.po_date, p.po_status,
    o.customer_id, o.order_date, o.region,
    l.product_id, l.quantity, l.unit_price, l.line_amount,
    -- Window function marks the first line per PO
    if(l.line_id = min(l.line_id) OVER (PARTITION BY l.po_id), 1, 0) AS is_first_line
FROM po_line AS l
JOIN purchase_order AS p ON l.po_id = p.po_id
JOIN order_table    AS o ON l.order_id = o.order_id;
```

**Query patterns:**

```sql
-- Header query — filter is_first_line to eliminate double-count
SELECT vendor_id,
    countIf(is_first_line = 1)         AS po_count,
    sumIf(po_total, is_first_line = 1) AS total_spend
FROM po_flat_c
WHERE po_date >= today() - 30
GROUP BY vendor_id;

-- Line query — direct SUM, no special handling
SELECT product_id, sum(line_amount) AS revenue
FROM po_flat_c
GROUP BY product_id;

-- Cross-grain — single table, direct filter + DISTINCT
SELECT count(DISTINCT order_id)
FROM po_flat_c
WHERE product_id = 42;

-- Multi-dimension analysis — all dimensions available without joins
SELECT
    toStartOfMonth(po_date)   AS month,
    region,                   -- from order-grain, pre-joined
    vendor_id,                -- from PO-grain
    product_id,               -- from line-grain
    sum(line_amount)          AS revenue,
    sumIf(po_total, is_first_line=1) AS po_value
FROM po_flat_c
GROUP BY month, region, vendor_id, product_id
ORDER BY month, revenue DESC;
```

**Why the `is_first_line` trick works reliably:**  
The `line_id` column is deterministic — for a given `po_id`, the minimum `line_id`
is always the same row (it doesn't change between runs or reloads). Using a window
function at ingest time makes the marker a stored physical column, not a computed
expression at query time. Once stored, `sumIf(po_total, is_first_line=1)` is simply a
columnar filter — as fast as any other WHERE condition.

## 7. Root Cause Analysis: Why Pattern A is Slow

### 7.1 The FINAL penalty

`ReplacingMergeTree` is designed for tables where the same primary key may be inserted
multiple times (e.g., CDC upserts). ClickHouse deduplicates by keeping only the row with
the highest `version` value — but this deduplication is **lazy**: it happens during
background merge operations, not immediately on insert. At any point in time, multiple
versions of the same row may exist in different data parts.

To get correct results, you must append `FINAL` to every query:

```sql
SELECT ... FROM po_line_a FINAL ...
```

`FINAL` forces ClickHouse to perform the merge at query time. This means:
- All data parts for the queried partition range are read
- Rows are merged in-memory using the ReplacingMergeTree merge algorithm
- Only the winning version per primary key is emitted to the query

At 100M rows across many parts, this is equivalent to a **full table merge scan** —
regardless of any WHERE filters. The query planner cannot skip parts based on primary key
ranges when FINAL is active because deduplication must see all versions.

**Benchmark evidence:**

```
Pattern A Q2:XL — rows_read: 100,880,000  query_duration: 7,415ms
Pattern C Q2:XL — rows_read: ~0 (cached)  query_duration:   268ms
```

A 27.7× difference on identical business logic, purely due to the engine choice.

### 7.2 The join cost at scale

Even with `full_sorting_merge` skipping the sort phase, joining 100M line rows to 10M PO
rows requires holding both streaming cursors simultaneously and merging on `po_id`. At
XL tier (Q5:A:XL), peak RAM reached **1,126 MB** and query time was **21,892ms**.

Pattern C eliminates this entirely — Q5 becomes a single-table scan with no join:

```sql
-- Pattern C Q5: single table, no join, 182ms at XL
SELECT toStartOfMonth(po_date), vendor_id, sum(line_amount)
FROM po_flat_c
GROUP BY 1, 2;

-- Pattern A Q5: join required, 21,892ms at XL
SELECT toStartOfMonth(l.line_date), p.vendor_id, sum(l.line_amount)
FROM po_line_a AS l FINAL
JOIN purchase_order_a AS p ON l.po_id = p.po_id
GROUP BY 1, 2
SETTINGS join_algorithm = 'full_sorting_merge';
```

## 8. NLP2SQL Compatibility Analysis

This section is specific to the Scorpio NLP2SQL agent use case and is the primary
decision driver for the recommendation.

### 8.1 Query generation complexity per pattern

| Capability required from LLM | Pattern A | Pattern B | Pattern C |
|---|---|---|---|
| Standard SQL syntax only | ❌ needs FINAL, dictGet | ❌ needs ARRAY JOIN | ✅ yes |
| Single table for most questions | ❌ 3 tables | ❌ 2 tables | ✅ 1 table |
| Direct SUM on revenue fields | ✅ | ✅ via ARRAY JOIN | ✅ |
| Direct SUM on header fields | ⚠ needs FINAL | ✅ | ⚠ needs is_first_line |
| JOIN knowledge required | ✅ always | ⚠ occasionally | ❌ never |
| Schema tokens per query | High (3 tables) | Medium (2 tables) | Low (1 table) |
| Risk of silently wrong results | High (forgot FINAL) | High (forgot ARRAY JOIN) | Low (forgot is_first_line gives 10× not wrong type) |

### 8.2 Failure modes by pattern

**Pattern A failure modes for NLP2SQL:**

```sql
-- LLM forgets FINAL → duplicate rows, overcounted aggregations
SELECT vendor_id, sum(po_total) FROM purchase_order_a  -- missing FINAL
GROUP BY vendor_id;
-- Result: silently wrong if any CDC duplicates exist; correct only if no duplicates

-- LLM omits SETTINGS → hash join used instead of merge join (10× slower)
SELECT ... FROM po_line_a JOIN purchase_order_a ...
-- Missing: SETTINGS join_algorithm = 'full_sorting_merge'

-- LLM doesn't know dictGet syntax
SELECT region FROM po_line_a  -- region doesn't exist on po_line_a
-- Error: unknown column. LLM must know to use dictGet('bench.order_dict', 'region', order_id)
```

**Pattern B failure modes for NLP2SQL:**

```sql
-- LLM uses direct SUM on array column → syntax error
SELECT sum(lines.line_amount) FROM purchase_order_b  -- missing ARRAY JOIN
GROUP BY product_id;
-- Error: cannot aggregate Array column without ARRAY JOIN

-- LLM uses WHERE instead of has() for array membership
SELECT * FROM purchase_order_b WHERE lines.product_id = 42;
-- Error or wrong result: cannot compare Array to scalar without has()

-- Standard SQL tools and BI connectors break on Array columns
SELECT lines.line_amount FROM purchase_order_b;
-- Result: returns Array type, not scalar — downstream tools fail
```

**Pattern C failure modes for NLP2SQL:**

```sql
-- LLM forgets is_first_line → double-counted header metrics
SELECT vendor_id, sum(po_total) FROM po_flat_c GROUP BY vendor_id;
-- Result: 10× overcounted (detectable: number is suspiciously large)

-- This is the ONLY failure mode, and it's self-correcting:
-- The overcounted result is obviously wrong (total spend = 10× gross revenue)
-- vs Pattern A where forgetting FINAL gives a plausibly correct but wrong answer
```

### 8.3 Prompt engineering implications

For NLP2SQL, the system prompt must include the schema and any special rules the LLM
must follow. The token cost and reliability of each pattern:

**Pattern A system prompt addition:**
```
Tables: order_a, purchase_order_a, po_line_a, order_dict
Rules:
- Always append FINAL to any query on order_a, purchase_order_a, or po_line_a
- For joins between po_line_a and purchase_order_a, use SETTINGS join_algorithm = 'full_sorting_merge'
- To get region or order_date from a line-level query, use: dictGet('bench.order_dict', 'region', toUInt64(order_id))
- po_total must only be summed from purchase_order_a, never from po_line_a
```

**Pattern B system prompt addition:**
```
Tables: purchase_order_b, order_b
Rules:
- Line-level columns (lines.product_id, lines.line_amount, etc.) require ARRAY JOIN lines before use
- To check if a PO contains a product, use: has(lines.product_id, <value>)
- Header columns (po_total, vendor_id, po_date) can be aggregated directly without ARRAY JOIN
- Joining to order_b for region requires: JOIN order_b USING (order_id)
```

**Pattern C system prompt addition:**
```
Table: po_flat_c
Rules:
- line_amount, quantity, unit_price: always safe to SUM directly
- po_total: use sumIf(po_total, is_first_line = 1) to avoid double-counting
- For order counts: use count(DISTINCT order_id), not count(DISTINCT po_id)
- All dimension columns (region, vendor_id, product_id) are available directly — no joins needed
```

Pattern C's prompt addition is the shortest, simplest, and most robust. The single rule
(`sumIf` for `po_total`) is easy for the LLM to apply consistently. Pattern A requires
four distinct rules, any of which can be forgotten or misapplied.

### 8.4 Dynamic SQL generation examples

Below are representative NLP2SQL translations for each pattern, showing how the LLM
must reason differently:

**User question**: *"What is our total procurement spend by region for Q1 2024?"*

```sql
-- Pattern A (LLM must remember: FINAL + dictGet for region)
SELECT
    dictGet('bench.order_dict', 'region', toUInt64(order_id)) AS region,
    sum(po_total) AS total_spend
FROM purchase_order_a FINAL
WHERE po_date BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY region
ORDER BY total_spend DESC;

-- Pattern B (LLM must join order_b for region)
SELECT
    o.region,
    sum(p.po_total) AS total_spend
FROM purchase_order_b AS p
JOIN order_b AS o ON p.order_id = o.order_id
WHERE p.po_date BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY o.region
ORDER BY total_spend DESC;

-- Pattern C (LLM writes standard SQL, region already in table)
SELECT
    region,
    sumIf(po_total, is_first_line = 1) AS total_spend
FROM po_flat_c
WHERE po_date BETWEEN '2024-01-01' AND '2024-03-31'
GROUP BY region
ORDER BY total_spend DESC;
```

**User question**: *"Which products were ordered in EMEA but not in APAC?"*

```sql
-- Pattern A (two separate scans on po_line_a FINAL + cross joins with dictGet)
SELECT DISTINCT l.product_id
FROM po_line_a AS l FINAL
WHERE dictGet('bench.order_dict', 'region', toUInt64(l.order_id)) = 'EMEA'
  AND l.product_id NOT IN (
      SELECT DISTINCT l2.product_id
      FROM po_line_a AS l2 FINAL
      WHERE dictGet('bench.order_dict', 'region', toUInt64(l2.order_id)) = 'APAC'
  );

-- Pattern C (single table, region is a plain column)
SELECT DISTINCT product_id
FROM po_flat_c
WHERE region = 'EMEA'
  AND product_id NOT IN (
      SELECT DISTINCT product_id FROM po_flat_c WHERE region = 'APAC'
  );
```

Pattern C's SQL is shorter, uses standard syntax, and requires no special knowledge of
ClickHouse-specific features.

## 9. Production Considerations

### 9.1 Storage cost of Pattern C

Denormalizing three tables into one means PO header fields are repeated on every line row.
For 10 lines per PO (our benchmark ratio):

| Field repeated | Type | Cost per line row | × 10 lines per PO |
|---|---|---|---|
| order_id | UInt64 | 8 bytes | stored once (ORDER BY) |
| vendor_id | UInt32 | 4 bytes | 40 bytes per PO |
| po_total | Decimal(18,2) | 16 bytes | 160 bytes per PO |
| po_date | Date | 2 bytes | 20 bytes per PO |
| po_status | LowCardinality | ~1 byte | ~10 bytes per PO |
| customer_id | UInt32 | 4 bytes | 40 bytes per PO |
| order_date | Date | 2 bytes | 20 bytes per PO |
| region | LowCardinality | ~1 byte | ~10 bytes per PO |

Rough additional storage vs normalised tables: ~300 bytes/PO × 10M POs = **~3 GB raw**,
**~0.8 GB compressed** (ClickHouse compresses repeated values extremely well — LZ4 on
repeated `po_total` and `vendor_id` values approaches 10× compression).

At XL tier (100M lines), the flat table occupied ~6 GB compressed vs ~2 GB for the
normalised Pattern A tables — a ~3× storage overhead, but still within a single
NVMe drive on any production server.

### 9.2 Write pipeline requirements

| Concern | Pattern A | Pattern B | Pattern C |
|---|---|---|---|
| CDC upsert model | ✅ `ReplacingMergeTree` handles naturally | ⚠ Must rebuild entire PO row | ✅ Insert new row with new version |
| Partial line arrival | ✅ Each table inserts independently | ❌ All lines must arrive together | ✅ Each line inserts independently |
| Line-level updates | ✅ ReplacingMergeTree per key | ❌ Requires full PO row rebuild | ✅ Insert new version per line_id |
| Out-of-order delivery | ✅ Version column handles it | ❌ Partial arrays break correctness | ✅ Version column handles it |
| ETL join required | ❌ No | ❌ No (but groupArray needed) | ✅ Yes — join at ingest |

Pattern C requires a join at ingest time (to attach order and PO attributes to each line).
This is the main write-side cost: the ETL pipeline must have access to all three source
tables simultaneously when inserting line rows.

For CDC pipelines (Debezium → Kafka → ClickHouse), the recommended approach is:
1. Maintain a small in-memory or Redis cache of `po_id → {vendor_id, po_total, po_date, po_status}`
2. Enrich each line event before writing to `po_flat_c`
3. On PO header updates, trigger a re-write of all line rows for that PO (or use a
   `ReplacingMergeTree` + `FINAL` view on top of `po_flat_c` if updates are frequent)

### 9.3 Pattern B's memory wall

| Tier | Pattern B insert peak RAM | Outcome |
|---|---|---|
| S (50K POs) | ~500 MB | Single INSERT, OK |
| M (100K POs) | ~1 GB | Single INSERT, OK |
| L (1M POs) | ~8 GB | Single INSERT, OK (just fits in 16 GB) |
| XL (10M POs) | >14 GB | **OOM crash** — requires 10 batched INSERTs |

This is not a configuration problem. The root cause is that `groupArray()` in the
aggregation phase must hold all array values for all POs simultaneously in the hash table.
At 10M POs × 10 lines × 5 columns × ~8 bytes = ~4 GB minimum, plus aggregation state
overhead, the actual peak exceeds container memory.

The XL workaround (10 batched INSERTs by `po_id` range) works but:
- Cannot be done in a streaming CDC pipeline without checkpointing
- Requires knowing the PO count in advance to size the batches
- Adds ~10× the ETL complexity vs Pattern C's single streaming INSERT

### 9.4 ClickHouse version notes

The `full_sorting_merge` join algorithm was introduced in ClickHouse 22.x. All results in
this study were obtained on **24.3.18**. If running on versions < 22.0, Pattern A's
cross-grain query performance would be significantly worse (hash join only).

The `ARRAY JOIN` syntax and `has()` function are available in all ClickHouse versions ≥ 1.x.

## 10. Decision Matrix

### 10.1 Weighted scoring

Weights reflect Scorpio's priorities: NLP2SQL reliability > query performance > write simplicity > storage.

| Criterion | Weight | Pattern A | Pattern B | Pattern C |
|---|---|---|---|---|
| NLP2SQL query generation simplicity | 30% | 3/10 | 4/10 | **9/10** |
| Query performance at scale (XL) | 25% | 2/10 | 7/10 | **9/10** |
| Memory efficiency | 15% | 3/10 | 8/10 | **10/10** |
| Write pipeline simplicity | 15% | 9/10 | 5/10 | 7/10 |
| BI tool compatibility | 10% | 9/10 | 3/10 | **9/10** |
| Schema simplicity (token budget) | 5% | 4/10 | 6/10 | **9/10** |
| **Weighted total** | **100%** | **4.2/10** | **5.8/10** | **9.0/10** |

### 10.2 Recommended decision

**Adopt Pattern C (flat denormalized at line grain) as the primary ClickHouse gold mart
design for procurement hierarchy data in Scorpio.**

For the NLP2SQL agent (`nlp2sql_agent`), the gold mart table should be a single flat table
following the `po_flat_c` schema with `is_first_line` as the double-count guard.

**Caveats and escape hatches:**

1. **If write pipeline is CDC with high update frequency on PO headers**: Consider a hybrid
   where `po_line_a` (Pattern A, `ReplacingMergeTree`) is the write target, and a
   materialized view populates `po_flat_c` (Pattern C) as the read target. This gives
   the write simplicity of Pattern A with the query simplicity of Pattern C, at the cost
   of one materialized view maintenance.

2. **If header-only dashboards are the primary use case**: A small `purchase_order_summary`
   `AggregatingMergeTree` materialized view on top of `po_flat_c` can provide
   sub-millisecond header aggregations by pre-computing `sumState(po_total)` per
   `(vendor_id, toYYYYMM(po_date))`.

3. **For the specific case of "does order contain product X" at very high QPS**:
   Pattern B's `has()` is marginally faster than Pattern C's `DISTINCT` (76ms vs 87ms
   at XL). If this specific query pattern is the dominant query at production scale,
   a separate `product_set_index` table using `AggregatingMergeTree` with
   `groupArrayState(product_id)` could be considered as a supplementary structure.

## 11. Implementation Roadmap

### Phase 1 — Adopt Pattern C for new gold marts (immediate)

All new procurement gold mart DDL should follow the `po_flat_c` schema:

```sql
-- Template for any future procurement gold mart
CREATE TABLE gold_procurement_mart (
    -- Temporal partition key (always first in ORDER BY)
    po_date         Date,
    -- Foreign keys (low-cardinality first in ORDER BY = better compression)
    vendor_id       UInt32,
    -- Grain keys
    po_id           UInt64,
    line_id         UInt32,
    -- Header attributes (denormalized)
    order_id        UInt64,
    customer_id     UInt32,
    order_date      Date,
    region          LowCardinality(String),
    po_status       LowCardinality(String),
    -- Line attributes
    product_id      UInt32,
    category_id     UInt32,
    quantity        UInt32,
    unit_price      Decimal(18,2),
    line_amount     Decimal(18,2),
    -- Double-count guard
    is_first_line   UInt8
) ENGINE = MergeTree()
  PARTITION BY toYYYYMM(po_date)
  ORDER BY (po_date, vendor_id, po_id, line_id);
```

### Phase 2 — Update NLP2SQL system prompt

The nlp2sql_agent system prompt should include:

```
You are generating ClickHouse SQL against the gold_procurement_mart table.

Column semantics:
- line_amount, quantity, unit_price → always safe to SUM directly (line-grain metrics)
- po_total → DO NOT SUM directly; use sumIf(po_total, is_first_line = 1)
- po_id + line_id → unique row identifier (line grain)
- order_id → parent order (40 lines per order on average)
- vendor_id, region, product_id → dimension columns, available directly

Do NOT use JOIN, ARRAY JOIN, FINAL, or dictGet. Everything is in one table.
```

### Phase 3 — Materialized view for pre-aggregated header metrics

```sql
CREATE TABLE procurement_header_agg (
    po_date         Date,
    vendor_id       UInt32,
    region          LowCardinality(String),
    po_count        AggregateFunction(count),
    total_spend     AggregateFunction(sum, Decimal(18,2))
) ENGINE = AggregatingMergeTree()
  PARTITION BY toYYYYMM(po_date)
  ORDER BY (po_date, vendor_id, region);

CREATE MATERIALIZED VIEW procurement_header_agg_mv
TO procurement_header_agg AS
SELECT
    po_date, vendor_id, region,
    countState()                             AS po_count,
    sumState(po_total)                       AS total_spend
FROM gold_procurement_mart
WHERE is_first_line = 1   -- only aggregate header rows
GROUP BY po_date, vendor_id, region;
```

This pre-aggregation makes dashboard Q1/Q4 queries sub-millisecond at any data volume.

### Phase 4 — Monitoring and alerting

Add to the observability dashboard:
- `system.query_log` alert: any query on procurement tables taking > 500ms
- `system.query_log` alert: peak memory > 1GB for any procurement query
- Weekly check: `SELECT count() FROM gold_procurement_mart WHERE is_first_line NOT IN (0,1)`
  (validates marker integrity after ETL runs)
